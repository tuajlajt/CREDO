AGENTIC WORKFLOW

CREDO

A virtual medical board for every patient.
Complete health profile in. Specialist consensus out.

Not a transcription tool — a clinical reasoning system. CREDO ingests the patient's complete health profile — medical history, lab trends, imaging, vitals, medications, and today's encounter — then convenes a panel of AI specialist agents who study the case, cross-reference findings, and deliver unified documentation with full chain-of-thought audit trails.

Universal Medical Image Encoder

MEDSIGLIP · 400M PARAMS · 448×448
Dual-tower vision+text encoder. Processes all medical image modalities into a shared embedding space: CXR, dermatology, ophthalmology, pathology, CT/MRI. Produces numerical embeddings — the same encoder inside MedGemma's vision system, but standalone is far more efficient when you only need quantitative comparison, not text generation.
IN ← current image(s) from Stage 01
IN ← matched prior image(s) from Stage 01
OUT → image embeddings per image (N × 768)
OUT → zero-shot classification scores
OUT → pairwise similarity (current vs prior)
OUT → progression drift score
OUT → change-point indices (longitudinal series)

Clinical Reasoning Agent — The Attending

MEDGEMMA 4B-IT (MULTIMODAL) · CHAIN-OF-THOUGHT
The AI primary care physician — the first to see the patient. Receives the complete health profile — not isolated data points, but the full patient context. Performs initial clinical reasoning with explicit chain-of-thought. Builds preliminary SOAP structure, extracts medication list, identifies clinical concerns.

S / O LOCKED TO TRANSCRIPT: The Subjective and Objective sections are derived exclusively from what the doctor and patient said — no inference, no fabrication from EHR or imaging. Only Assessment (A) and Plan (P) synthesise across the full health profile and specialist findings.

CONVENES THE BOARD: Based on all available evidence — imaging modalities, transcript findings, lab abnormalities, patient history, medication signals — decides which specialist agents to consult, and which are not needed. Just as a real attending would say "I want cardiology and pulm to look at this," the orchestrator routes the case to only the relevant specialists.
IN ← clinical transcript (from MedASR)
IN ← full patient EHR context
IN ← active medication list with dosages
IN ← current + historical labs & vitals
IN ← MedSigLIP embeddings + classification + drift scores
OUT → preliminary SOAP — S / O locked to transcript; A / P synthesised (with CoT)
OUT → extracted medication list
OUT → preliminary ICD-10 candidates
OUT → clinical concern flags
OUT → board routing → specialist(s) to consult
OUT → tailored context packet per specialist
The board convenes — specialist agents + medication checks dispatched simultaneously

Clinical Synthesis Agent — Board Summary

MEDGEMMA 27B-IT (TEXT) · CHAIN-OF-THOUGHT
A separate agent from the attending — think chief of staff reviewing all consult notes. Receives all specialist reports, medication alerts, and the preliminary SOAP from the attending (Stage 03). Synthesizes into final unified documentation with explicit chain-of-thought. Resolves cross-specialty conflicts — when the radiologist sees an infiltrate and the medication checker suspects drug-induced pneumonitis, the synthesis agent weighs both perspectives against the evidence and documents the reasoning.

PHYSICIAN REVIEW + EDIT: The doctor reviews the synthesized output here and can edit any section — modifying the SOAP note, adjusting ICD-10 codes, adding clinical context, or overriding agent recommendations. All physician edits are tracked as part of the audit trail. When the doctor prescribes new medications → Stage 06 fires automatically.

🔀 Board Routing Logic (Stage 03)

The attending decides which specialists to consult based on:

• Imaging present: CXR → radiologist; skin photo → dermatologist; fundus → ophth

• Transcript signals: "rash on arms" → dermatologist even without image

• Lab triggers: elevated troponin → cardiologist; high HbA1c → endocrinologist

• History patterns: known eczema + new wheeze → derm + pulm (atopic march)

• Cross-specialty: diabetic + fundus image → endocrinologist + ophthalmologist

• Always active: Medication Symptom Checker + DDI Engine run on every visit

⛓️
Chain-of-Thought at every stage. The Attending (03), every Board Specialist (04), the Medication Symptom Checker (04), and the Synthesis Agent (05) all produce explicit reasoning traces. Combined with tracked physician edits, this creates a complete audit trail: what the system recommended, why, and what the physician accepted, modified, or overrode. This documentation structure is designed to support the transparency and traceability that regulatory pathways (such as FDA Software as a Medical Device frameworks) and clinical liability review would expect — though actual regulatory compliance depends on the specific deployment context and approval process.
